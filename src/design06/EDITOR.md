# design06 — соглашения для будущего редактора

Эти правила выбраны совместно и фиксируют, как устроен слой стилей и как над ним
строится визуальный редактор (move/resize/rotate, добавление текста/картинок,
типографика). Главный инвариант — **0%**: при пустых правках рендер пиксель-в-пиксель
совпадает со статичным эталоном `public/design06-exact/`.

## 1. Где живут стили (Approach A2)

- **База (структура + поведение)** — JSX-разметка секции + утилиты-классы Canva в
  `<Section>.module.css` (`position:absolute`, flex, `object-fit`, `:hover`, тема,
  псевдоэлементы, комбинаторы). Это неизменяемый фундамент, даёт 0%.
- **Редактируемый слой** — на элемент одна запись `El` в `<Section>.layout.ts`,
  ключ = `data-eid`. Применяется инлайном: `style={elStyle(layout[eid])}`.
- Тип `El` и сборщик `elStyle` — в [`layout.ts`](./layout.ts):
  - геометрия: `x, y, w, h, rot, scale`;
  - текст: `font, fontSize, letterSpacing, lineHeight, textAlign, textTransform, color`
    (в Canva размер текста = CSS-переменная `--H97cbQ`, `fontSize` маппится на неё);
  - `raw` — всё не разложенное (вторичные custom-props, сложные `transform` масок и т.п.).
- Утилиты-классы и инлайн складываются: класс = «какой это объект и как себя ведёт»,
  `style`-из-записи = «где он и какой именно». Инлайн имеет высшую специфичность.

## 2. Контракт data-eid ↔ запись

- На любом элементе **`data-eid` ВСЕГДА равен ключу, который он читает** в
  `style={elStyle(layout[eid])}`. Переименование — синхронно в `.tsx` и `.layout.ts`.
- Рантайм-проверка: число `data-eid` в DOM == числу ключей `layout`, без дублей.
- Имена `data-eid` — семантические (`hero/title`, `calendar/grid`, `timeline/divider-1`),
  без голых чисел.

## 3. Что выбираемо в редакторе (решение B)

`data-eid` присутствует только на том, что редактор адресует. **Чистые обёртки и
пустые spacer/overlay-боксы `data-eid` НЕ несут** (их запись/стиль остаются — 0% цел —
но кликнуть их нельзя).

- **Клик-выбираемые БЛОКИ** (то, что двигаешь/ресайзишь): `DF_utQ`-блоки, картинки
  (обёртка-кроп `Izwocg`), корень секции (`rGeu6w`).
- **Текст НЕ отдельный таргет.** Выбирается блок; панель показывает две группы:
  - **«Поле»** — из записи блока: `x, y, w, h, rot` (ширина → перенос строк/wrap; габариты; поворот);
  - **«Текст»** — из вложенной записи `<p>`: `font, fontSize, letterSpacing, lineHeight, color`.
  - Редактор по выбранному блоку сам находит вложенную текст-запись `<p>`.
- **Внутренние элементы** (цепочки `Zp7NQw›a26Xuw›PcHy7w›uk_25A`, scale-wrap, пустые box,
  отдельные span/ячейки) — без `data-eid`, не выбираемы.

### Resize: два разных «размера»
- **Размер поля** (`w/h` блока) — габариты контейнера: ширина управляет переносом строк,
  глифы не меняет.
- **Размер шрифта** (`fontSize` → `--H97cbQ` в записи `<p>`) — кегль букв.
- Нюанс: часть заголовков отрисована со `scale()` на внутренней обёртке (напр.
  «Felix & Angel» — `scale(1.46)`), поэтому видимый размер = `fontSize × scale`.
  В редакторе — показывать «эффективный размер» (перемножив) либо при правке `fontSize`
  сворачивать `scale` в 1.

## 4. Структуру НЕ упрощаем (обёртки оставляем)

- Глубокая вложенность Canva (один блок-картинка ≈ 14 уровней) — цена 0% (точный
  кроп/клип/масштаб). Не схлопываем.
- Редактор оперирует **БЛОКАМИ** (семантическими под-компонентами: `HeroFrame`,
  `ColorSwatch`, `Divider`…), обёртки инкапсулированы внутри блока и редактору не видны.

## 5. copy / add / delete — операции над данными, не над DOM

- **Copy** — клонировать под-компонент-блок + его записи под новыми `eid` (с неймспейсом);
  обёртки копируются сами. Параметризованные (`<Divider eid=…/>`) — тривиально.
- **Add** — новый элемент из **простого шаблона** (`<div absolute style={record}><img/text></div>`),
  НЕ из 14-уровневой Canva-цепочки.
- **Delete** — убрать инстанс блока + его записи.

## 6. Сохранение и откат

- **Save** пишет ровно в `<Section>.layout.ts` (структурный патч объекта по ключу) —
  без AST-боли. Для браузер→файл — dev-эндпоинт (Vite-плагин), см. `vite-plugins/`.
- **Откат** = git: закоммиченный `layout.ts` — это **база (не эталон)**, стартовые 0%.
  Правки идут поверх коммита; `git checkout <file>` или возврат на коммит базы = откат.
  Эталон для пиксель-сверки — статичная копия `public/design06-exact/`, а не `layout.ts`.
- **Генераторы** (`tools/design06-*.mjs`) запускаются вручную, один раз; редактор,
  сборка и dev-сервер их НЕ вызывают, перезатереть правки они не могут.

## 7. Проверка после любой правки

- Открыть `/?d06` (адаптивно: ≤880 — подгон под ширину, >880 — фикс 880 по центру)
  или `/?d06&noscale` (1:1 для пиксельной сверки).
- Пиксель-дифф против `design06-exact` должен быть **0.000%** при пустых правках.

## Файлы

- `layout.ts` — `El` + `elStyle`.
- `sections/<S>.tsx` — структура + утилиты-классы (база) + применение по `data-eid`.
- `sections/<S>.layout.ts` — редактируемые записи (источник правды для редактора).
- `sections/<S>.module.css` — утилиты-классы.
- `cx.ts` — сборка классов.
- Генераторы: `tools/design06-{extract,gen-sections,css-modules,layout-data,name-eids,mark-selectable}.mjs`.
